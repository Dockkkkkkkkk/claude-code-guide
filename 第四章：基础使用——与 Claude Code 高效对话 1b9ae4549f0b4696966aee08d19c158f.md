## 4.1 四种交互模式

### 交互式对话（最常用）

```bash
claude
```

启动后进入持续对话模式，Claude 会记住本次会话的上下文。

### 单次执行模式

```bash
# -p 参数直接执行任务，无需进入交互界面
claude -p "帮我把 data.csv 里的数据做统计分析"

# 适合脚本调用或快速一次性任务
claude -p "检查 app.py 是否有语法错误"
```

### 管道模式（Unix 风格）

```bash
# 把文件内容通过管道传给 Claude
cat 报告草稿.txt | claude -p "帮我润色这段文字，保持专业风格"

# 处理命令输出
git diff | claude -p "帮我写一条简洁的 commit message"

# 批量处理
ls *.py | claude -p "分析这些 Python 文件，找出潜在问题"
```

### 无头/CI 模式（用于自动化脚本）

```bash
# --headless 或 --no-interactive 适合在 CI/CD 中运行
claude --headless -p "检查当前目录代码，修复所有 lint 错误"

# 指定额外工作目录
claude --add-dir /path/to/extra/dir
```

## 4.2 Think 模式——让 AI 先想清楚再动手

Claude Code 支持不同深度的思考模式，通过在提示中加入关键词触发：

| **模式** | **触发方式** | **Token消耗** | **适用场景** |
| --- | --- | --- | --- |
| 普通模式 | 直接输入任务 | 低 | 简单问答、小修改 |
| Think | 提示中加「think」 | 中 | 需要一定思考的任务 |
| Think Hard | 提示中加「think hard」 | 高 | 复杂规划、多步骤任务 |
| Ultrathink | 提示中加「ultrathink」 | 极高 | 高难度问题、架构设计 |

**使用示例：**

```
# 普通任务
帮我修复第23行的拼写错误

# 需要思考的任务
think about how to refactor this authentication module

# 复杂架构设计
think hard: 设计一个支持百万用户的消息推送系统架构

# 最高难度
ultrathink: 分析这个项目的所有技术债，制定6个月的重构计划
```

## 4.3 @ 文件引用与 1M 上下文的正确用法

### @ 语法引用文件

```
# 引用单个文件
@README.md 帮我更新这个文档，加入最新的安装说明

# 引用多个文件
@src/auth.py @src/models.py 这两个文件有什么耦合问题？

# 引用整个目录
@src/ 分析这个目录下所有文件的代码质量
```

### 1M Token 上下文的正确使用方式

- **适合塞入 1M 上下文的内容**：整个代码仓库、完整文档、大型数据集
- **不要浪费上下文**：无关文件不要引用，会增加成本和降低效果
- **会话过长时使用 `/compact`**：Claude 会自动压缩旧上下文保留关键信息

```bash
# 将整个项目目录作为上下文
claude --add-dir ./my-project

# 对话中手动压缩
/compact
```

## 4.4 Auto-Plan——自动规划模式（2026年新功能）

Auto-Plan 让 Claude 在执行复杂任务前**先列出执行计划**，由你审查确认后再动手。

**效果对比：**

- 无 Auto-Plan：Claude 直接开始执行，可能走偏方向
- 有 Auto-Plan：Claude 先展示计划 → 你确认 → 再执行

**触发方式：**

```
# 在任务描述中加入触发词
请先制定计划，然后帮我重构整个 auth 模块

# 或者使用命令
/plan 帮我把这个 Flask 项目迁移到 FastAPI
```

**最佳使用时机：**

- 涉及多个文件修改的任务
- 不确定 AI 会怎么做的复杂任务
- 有风险的操作（删除、重构、迁移）

## 4.6 Voice Mode——语音编程（2026年新功能）

Voice Mode 让你可以直接**说话**给 Claude Code 下达任务，无需打字。

**启用方式：**

```bash
# 在交互式模式中按快捷键启动语音输入
# macOS: Cmd + Shift + V
# Windows: Ctrl + Shift + V
```

**最佳使用场景：**

- 双手不空闲时（边思考边移动）
- 描述复杂需求时（说话比打字更自然）
- 长任务描述（语音识别比打字快3-5倍）

**Voice Mode + Think Hard 组合：**

```
# 语音输入：
"think hard，帮我设计一个适合小型团队的代码审查流程，
要考虑到我们是远程团队、每周发布一次、团队有5个人的情况"
```

## 4.7 权限控制与安全

### 三级权限模型

Claude Code 在执行操作前会请求权限：

- **读取权限**：读文件、搜索目录（默认允许）
- **写入权限**：创建/修改文件（会提示确认）
- **执行权限**：运行命令/脚本（会显示命令内容请求确认）

### 安全使用最佳实践

```bash
# 高危操作前，先用 --dry-run 预览（如果工具支持）
claude -p "预览一下如果删除 temp/ 目录会影响哪些文件，但不要真的删除"

# 设置工作目录限制（只允许操作指定目录）
claude --add-dir ./safe-directory

# 在 CLAUDE.md 中明确禁止的操作
# 例如：禁止直接推送到 main 分支、禁止删除数据库等
```

### 审查技巧

- 执行危险命令前，先问「你打算怎么做？列出步骤」
- 涉及生产环境操作，明确告诉 Claude「这是生产环境，请谨慎」
- 不确定的操作可以先在测试目录验证

## 🔬 实战案例

### 案例1：用 Voice Mode 完成产品需求文档

```
# 语音说：
"帮我写一份产品需求文档，
产品是一个面向中小企业的AI客服系统，
主要功能包括：自动回复、人工转接、对话历史分析。
目标用户是企业客服团队负责人，他们最关心的是：
减少人力成本、提升响应速度、方便统计分析。
文档格式按照标准PRD模板，保存到 PRD.md"
```

### 案例2：用 Auto-Plan 重构复杂模块

```
请先制定详细计划，然后重构 src/payment/ 目录下的支付模块：
1. 统一错误处理方式
2. 提取公共的 retry 逻辑
3. 添加完整的日志记录
4. 不改变对外接口

先告诉我你的重构计划，我确认后再执行。
```

### 案例3：管道模式批量处理 Markdown 文件

```bash
# 批量润色所有 Markdown 文档
for file in docs/*.md; do
  cat "$file" | claude -p "润色这篇技术文档，保持专业风格，修正语法错误" > "${file%.md}_revised.md"
done
```
