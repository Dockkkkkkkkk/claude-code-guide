# 第五章：CLAUDE.md——打造专属 AI 助手

# [CLAUDE.md](http://CLAUDE.md)——打造专属 AI 助手 🟡

> **核心价值**：[CLAUDE.md](http://CLAUDE.md) 是 Claude Code 的「长期记忆」，让每次对话都从正确的起点开始。
> 

---

## 5.1 [CLAUDE.md](http://CLAUDE.md) 是什么？为什么重要？

**没有 [CLAUDE.md](http://CLAUDE.md) 的情况：**

每次启动 Claude Code，它对你的项目一无所知。你要反复解释技术栈、代码规范、禁止事项……

**有 [CLAUDE.md](http://CLAUDE.md) 的情况：**

Claude Code 每次启动自动读取这个文件，立刻了解你的项目背景、偏好和规则，直接开始工作。

本质上，[CLAUDE.md](http://CLAUDE.md) 就是你给 AI 助手写的**入职手册**。

---

## 5.2 三层加载层级与优先级

```
全局：~/.claude/CLAUDE.md          ← 所有项目通用设置（你的个人偏好）
  └── 项目：./CLAUDE.md             ← 当前项目专属配置
       └── 子目录：src/CLAUDE.md    ← 特定模块专属规则
```

**优先级规则**：越靠近当前目录的配置，优先级越高。

**使用建议**：

- `~/.claude/CLAUDE.md` —— 写你的个人习惯（输出语言、回复风格）
- 项目根目录 `CLAUDE.md` —— 写项目技术栈、规范、禁止事项
- 子目录 `CLAUDE.md` —— 写该模块特有的规则

## 5.3 完整结构模板（可直接复用）

```markdown
# 项目说明
项目名称：[项目名]
项目背景：[一句话描述这个项目是什么，解决什么问题]
当前阶段：[开发中 / 测试 / 生产维护]

# 技术栈
- 语言：Python 3.11
- 框架：FastAPI 0.104
- 数据库：PostgreSQL 15 + Redis 7
- 前端：Vue 3 + TypeScript
- 部署：Docker + Kubernetes

# 工作规范
- 代码风格：遵循 PEP 8，使用 Black 格式化
- 命名规范：变量小写下划线，类名大驼峰
- 注释：重要逻辑必须有注释，使用中文
- 提交规范：feat/fix/docs/refactor 前缀

# 禁止事项
- 禁止直接推送到 main/master 分支
- 禁止硬编码任何密钥、密码、Token
- 禁止删除 migrations/ 目录下的文件
- 禁止修改生产环境的数据库 Schema

# 文件结构
src/
  api/          ← API 路由层
  services/     ← 业务逻辑层
  models/       ← 数据模型
  utils/        ← 工具函数
tests/          ← 测试文件（与 src/ 结构对应）

# 我的偏好
- 输出语言：中文
- 回复风格：简洁直接，不要废话
- 代码注释：关键部分加中文注释
- 遇到不确定的事情：先询问，不要自己假设
```

## 5.4 Auto-Memory——自动记忆（2026年新功能）

Auto-Memory 让 Claude 在对话中**自动发现重要信息并更新 [CLAUDE.md](http://CLAUDE.md)**，无需手动维护。

**工作原理：**

- Claude 发现新的重要信息（如你提到「我们用 Jira 管理需求」）
- Claude 自动询问：「是否要将这条信息保存到 [CLAUDE.md](http://CLAUDE.md)？」
- 你确认后，Claude 自动更新文件

**手动触发：**

```bash
# 在对话中命令 Claude 记住某件事
记住：我们的测试环境数据库地址是 test.db.internal:5432

# 或使用命令
/memory 记录：项目负责人是李明，微信 liming_dev
```

**手动 vs 自动的建议：**

- 项目核心规范：手动写入（保证精确性）
- 对话中临时发现的信息：让 Auto-Memory 自动记录
- 定期检查 [CLAUDE.md](http://CLAUDE.md)，清理过时信息

## 5.5 Rules 规则文件

`.claude/rules/` 目录可以存放模块化的规则文件，避免 [CLAUDE.md](http://CLAUDE.md) 过于臃肿。

```
项目根目录/
└── .claude/
    ├── CLAUDE.md          ← 主配置文件
    └── rules/
        ├── security.md    ← 安全相关规则
        ├── api-design.md  ← API 设计规范
        └── testing.md     ← 测试规范
```

[**security.md](http://security.md) 示例：**

```markdown
# 安全规则

## 必须遵守
- 所有用户输入必须经过验证和过滤
- SQL 查询必须使用参数化查询，禁止字符串拼接
- 敏感信息必须加密存储，禁止明文
- API 接口必须有认证，禁止匿名访问核心接口

## 代码审查要点
- 检查是否有硬编码的密钥
- 检查是否有未处理的异常导致信息泄露
- 检查日志中是否打印了敏感信息
```

## 🔬 实战案例

### 案例1：环境评估项目的 [CLAUDE.md](http://CLAUDE.md)

```markdown
# 建设用地土壤污染评估系统

项目背景：面向环境检测机构的专业评估报告生成系统
当前阶段：生产运营

# 技术栈
- 语言：Python 3.11
- 框架：Django 4.2 + DRF
- 数据库：PostgreSQL 14
- 报告生成：python-docx + reportlab

# 领域知识
- 评估标准：HJ 25.3-2019、GB 36600-2018
- 污染物单位：mg/kg（土壤）、mg/L（地下水）
- 风险评估计算公式在 src/calculation/formulas.py 中

# 规范
- 所有污染物名称使用标准中文名称
- 计算结果保留4位有效数字
- 报告模板在 templates/ 目录，不要随意修改格式

# 禁止
- 禁止修改已有的计算公式（需要走审批流程）
- 禁止在报告中使用非标准单位
```

### 案例2：个人全局 [CLAUDE.md](http://CLAUDE.md)

```markdown
# 我的个人偏好

## 输出风格
- 语言：中文
- 回复：简洁直接，结论放最前面
- 代码：关键逻辑加中文注释
- 不要：废话、客套、过度解释

## 技术偏好
- 优先使用标准库，非必要不引入新依赖
- 函数要有类型注解
- 错误要有明确的错误信息

## 工作习惯
- 遇到不确定的需求，先问我，不要自己假设
- 修改重要文件前，先告诉我你打算怎么改
- 完成任务后，简单说明做了什么、有什么注意事项
```

### 案例3：团队统一 [CLAUDE.md](http://CLAUDE.md)（Git 管理版）

```bash
# 在项目根目录创建 CLAUDE.md 并提交到 Git
# 让所有团队成员使用一致的规范

git add CLAUDE.md
git commit -m "docs: 添加 Claude Code 项目配置"
git push
```

> 团队共享 [CLAUDE.md](http://CLAUDE.md) 的好处：所有人使用 Claude Code 时都遵循同一套规范，避免 AI 给不同人产出风格迥异的代码。
